# エビデンスに基づく目標達成提案（ARIS）

> Bossの要望: **「こうすれば目標達成に効く」→「だからこう進めたい」**を、**エビデンス付き**で提案する。  
> 権限の境界は [board.md](../governance/board.md) と [DECISION-009](../decisions/decision_log.md) に従う。

---

## 1. 提案に必ず含めるブロック

| ブロック | 書くこと |
|----------|----------|
| **結論** | 何をするか（1〜3文）。 |
| **対応する目標** | `vision.md`・`reverse_path_300b.md` のどのレバー／フェーズに効くか。 |
| **因果の筋** | なぜその行動が企業価値・売上・リスクのどれを動かすか（論理のチェーン）。 |
| **エビデンス** | 下表の**種類**を明示し、**出典**を書く（ファイルパス・URL・日付・計測名）。 |
| **反証・限界** | うまくいかない条件、自分たちの前提の弱さ。 |
| **検証方法** | 次の四半期／30日で見る**指標**と、**見送りライン**（やめる条件）。 |
| **Bossへの依頼** | 承認が要る事項だけを列挙。 |

---

## 2. エビデンスの種類と強さ（目安）

| 種類 | 例 | 強さの目安 |
|------|-----|------------|
| **社内計測** | 売上、継続率、PoC結果、`valuation_log`、日次反省ログ | 最優先で積み上げる |
| **顧客・ユーザー** | インタビュー要約、失注理由、導入企業の数値 | 高 |
| **市場・競合** | 公開情報、価格帯、類似プロダクトの動き | 中〜高（出典必須） |
| **アナロジー** | 他社・他業界の類似事例 | 中（適用条件を書く） |
| **文献・標準** | セキュリティ標準、労務・個人情報の枠組み | 高（領域による） |

**禁止に近いこと**: 出典のない断定、「なんとなく」「業界では一般的」だけの主張。

---

## 3. 提出のしかた

- **軽い提案**: `communications/boss-aris/` に日付入りMarkdownでよい。  
- **方針に効く提案**: `decisions/decision_log.md` に採択時に追記、または [DECISION-XXX] として残す。  
- **繰り返し使う仮説**: `strategy/` または `projects/` にリンク。

---

## 4. ARISの自戒

- エビデンスが薄いときは**提案のタイトルに「仮説」**と明記し、**取れるデータ**を先に書く。  
- **日次反省**（`/ARIS/daily/`）で仮説が外れたら、次の提案に反映する。
