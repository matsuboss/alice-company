# ARIS PROJECT — 商品化と価格（ARIS提案＋エビデンス）

> 様式: [evidence_based_proposals.md](./evidence_based_proposals.md)  
> **結論**: 初年度は **「パイロットは企業あたり固定額」＋「本契約はサブスクリプション（継続課金）」** のハイブリッド。**請求軸は §0、サブスク方針は §0.1**。  
> 以下の金額は **仮置き** — 初契約・失注理由で [DECISION] または四半期レビューで更新する。

---

## 0. 請求の主軸（Boss方針 — 確定）

**PC台数・エンドポイント台数を請求の根拠にしない。**

| 理由 | 内容 |
|------|------|
| **収益と効果が逆相関しうる** | 今後効率化が進み **PCや端末が減る** と、台数課金では **当社収益だけが下がる**。顧客の成功とビジネスモデルが噛み合わない。 |
| **倹約行動を誘発する** | **1台に寄せて使う**などの回避が起き、**利用実態と対価**がずれる。 |

代わりに、**企業（または契約単位）あたりの月額・年額**、または **従業員数・組織規模の帯** など、**端末台数と連動しない**軸で設計する（詳細は下表・パイロット節）。

---

## 0.1 サブスクを主軌道にする（Boss方針）

**本番の収益の芯は「サブスク（月額または年額の継続課金）」とする。** §0 の請求軸で **MRR／年額換算（ARR）** を積み上げる。

| 理由 | 内容 |
|------|------|
| **予測可能なARR** | 投資・採用・インフラの計画が立てやすい。 |
| **顧客との長期関係** | プロダクト改善・Task Diary学習価値と相性がよい。 |
| **解約率の管理** | 単発売切りより **継続率・拡張（アップセル）** を経営指標にできる。 |

**運用**: **月額**を標準とし、**年払いは割引**（キャッシュフローと引き換え）でよい。エンプラは **年額＋最低保証**（すでにサブスクの一形態）。パイロットのみ **期間固定の前払い**（オンボーディング用）。

**その他の収益**（導入支援・スポット開発・アドオン等）は **サブスクを補完**し、**サブスクの解約を減らす・単価を上げる**方向に寄せる（§1.2）。

## 1. 結論（推奨パッケージ）

| フェーズ | 売り方 | ARISが「売れる」と判断する金額の芯 | 想定顧客 |
|----------|--------|-------------------------------------|----------|
| **パイロット（0〜3ヶ月）** | **固定額（3ヶ月）** | **30万〜80万円 / 企業（契約単位）/ 期間**（**対象範囲**は「組織・従業員レンジ」等で定義。**PC台数上限は請求根拠にしない**） | 初導入・稟議が早い中小〜中堅 |
| **本契約（標準）** | **サブスク — 企業あたり月額**（**従業員数帯**または**全社サイトライセンス**） | **MRRのレンジを帯で宣言**（例: 小規模〜中規模で **数万円〜数十万円/月** — 実額は原価・支援工数と整合して決定） | 本番運用・**継続更新** |
| **エンタープライズ** | **サブスク — 年額＋SLA** | **最低年額 300万〜800万円**（**年契約の前払い**）＋オプション | 大企業・規制業界 |

**推奨は1本**: まず **「パイロット 49万円（税別）/ 3ヶ月」**（対象は契約書で **法人・事業所スコープ・従業員レンジ** 等に定義）＋ **本契約は「企業あたり月額（帯）」** で提示し、初年度は **2〜3社の有償パイロット** で実績を取る。**980円/台/月のような単価表は採用しない。**

## 1.1 価格体系の細分化（運用イメージ）

### パイロット — 企業あたり固定（3ヶ月）

| 項目 | 内容 |
|------|------|
| **期間** | 契約開始日から **90日（3ヶ月）** を1期間とする。 |
| **第一候補** | **49万円（税別） / 3ヶ月**。**対象範囲**は契約で定義する（例: **特定法人の従業員〜名まで**、**本社＋指定拠点**）。**エージェントを入れる端末数で増減課金しない**。 |
| **レンジの使い分け（30万〜80万）** | **30万前後**: PoC中心・**対象組織が小さい**・オンラインミーティング中心。**49万**: 標準。**60〜80万**: オンバウンド・セキュリティレビュー同席・**導入キックオフ現地**など工数が乗る案件。 |
| **49万に含めるもの（例）** | エージェント配布用パッケージ、接続設定支援、**週1回×12回**の30分定例、パイロット終了時の**成果レポート（1冊）**、成功基準の事前合意。 |
| **含めない／別料金（例）** | **契約スコープ外法人の追加**、カスタム開発、24時間コール、**社内規程ドラフトの弁護士レビュー**（※**PC台数の超過課金は設けない**方向）。 |
| **支払い条件（例）** | 着手前 **100%前払い**（小額契約のキャッシュフロー優先）または **50%＋終了時50%**。 |
| **パイロット終了時** | **本契約へ移行**（下表の**企業あたり月額**） or **延長**（月額15〜25万/月の短期延長など別料金表） or **終了**（アンインストール手順の提供まで）。 |

---

### 本契約 — 企業あたり月額（従業員数帯など・PC台数は請求に使わない）

**定義**: 請求は **「契約企業（またはグループ内対象法人）あたりの月額」**。金額は **従業員数レンジ・売上規模帯・サイトライセンス** 等で帯設計し、**登録PC数・端末数とは連動させない**（効率化で端末が減っても当社の請求根拠が自動では毀損しない）。

| 規模の目安（例） | 月額の考え方（税別・仮置き） | 備考 |
|------------------|------------------------------|------|
| **小規模**（従業員〜30名程度まで 等） | **月額 5万〜15万円** 帯 | 最低利用料・サポート原価を回収。 |
| **中規模**（〜100名程度 等） | **月額 15万〜40万円** 帯 | 稟議は **総額** で説明。 |
| **準大企業**（100名超〜） | **個別見積** または **エンプラ年額**へ | 下表のエンタープライズと接続。 |

**最低利用料（例）**: 月額 **5万円から**（規模が小さくても運用・サポートの下限を確保）。

**請求サイクル**: 月次後払い or **年一括前払い（割引）** などは別紙。

**技術メモ**: 社内では端末を識別してもよいが、**課金の単位にしない**（表示・運用目的と課金目的を分離）。

---

### エンタープライズ — 年額＋SLA（最低年額 300万〜800万円）＋オプション

**前提**: 月額の「企業あたり帯」だけでは稟議・法務・情シスが通りにくい組織向け。**年額契約（最低保証）**で **SLAと窓口をセット**にする。**席数・PC台数ではなく、組織スコープと最低年額**で締める。

| 最低年額（税別） | 想定規模・中身のイメージ |
|------------------|---------------------------|
| **年額 300万円台（最低ライン）** | **中堅組織**向け。**SLA: 営業日9-18時**応答、可用性目標 **99.0%**（定義は別紙）、**専用チャット**1窓口。 |
| **年額 500万円前後** | **中堅エンプラ**。**SLA: 営業日 8-20時**、**優先オンコール**（重大インシデント）、**四半期レビュー**同席。 |
| **年額 800万円前後** | **大企業・規制業界**に近い。**SLA: 24×7の一次受付**（または時間外エスカレーション）、**専任CS**の名前付き、**ログ保管期間延長**、**稟議・RFPへの回答テンプレ拡充**。 |

**オプション（例・別途見積）**

| オプション | 内容の例 |
|------------|----------|
| **オンプレ／VPC内接続** | データの出ない構成、VPN要件— **+100万〜/年** から別見積。 |
| **SI・連携開発** | API連携、ID基盤連携— **プロジェクト単価**。 |
| **ログ保管延長** | 12ヶ月→36ヶ月保管など— **保管量×円/GB/月**。 |
| **カスタムレポート** | 経営会議向け四半期レポート— **年額固定アドオン**。 |

**契約スコープ**: エンプラでは **「年額最低保証に含まれる対象法人・事業所・従業員レンジ」** を**契約書で明確化**する（**端末台数ベースの増減は原則としない**）。

## 1.2 サブスク以外の収益（併走・補完）

**主軸は §0.1 の継続課金。** 以下は **ARPU を上げる・導入を早める** ための補助線。詳細な案のプールは [revenue_opportunities.md](./revenue_opportunities.md)。

| 区分 | 内容 | サブスクとの関係 |
|------|------|------------------|
| **プロフェッショナルサービス** | 導入キックオフ・情シス伴走・定額パッケージ（別枠請求） | 初回の **時間短縮** と **稟議通過** |
| **チャネル** | MSP・SIへの **紹介料／再販マージン** | **MRR の獲得チャネル**（直販の補完） |
| **アドオン（継続）** | 経営向けレポート拡張・ログ保管延長・カスタムレポート年額 | **サブスク上乗せ** で ARPU 向上 |
| **スポット** | 連携開発・自動化スクリプトの **請負** | 単発だが **エンジン活用**・ロックインに寄与しうる |
| **買い切り（限定的）** | 極小規模・特殊要件のみ **例外的に相談** | 原則はサブスク。**例外ルール**は `意思決定/` で |

**注意**: スポット・コンサルだけが肥大化すると **スケールしない**。売上構成比は **四半期レビュー** で見て、**MRR+年額換算（ARR）が主役**であることを維持する。

## 2. なぜこのレンジか（因果）

1. **市場アナロジー**  
   国内には **PC台数・ユーザー単位の月額** 製品が多く、比較説明の**参考**にはなる。ただし **当社の請求の主軸はそれにしない**（§0）。  
2. **ARIS PROJECT の深度**（操作リズム・ブラウザ文脈・サーバ連携・スクリプト実行）は、単純な勤怠・1機能だけのツールより**付加価値が高い**ため、**過度に安い単価帯では売らない**方がブランドと継続率を守れる。  
3. **B2Bの初年度**は、継続課金の総額が読めず稟議が止まりがち → **パイロットは企業あたり固定額**で **総額の明確化**と **成功基準の合意**を取りにいく。  
4. **§0 の方針** — 効率化の成果が **端末削減** として出たときも、**顧客の成功と収益モデルが両立**するよう、課金を **端末数から切り離す**。

## 3. エビデンス

### 3.1 国内の「PC・ユーザー単位・月額」の公開例（アナロジー・当社は請求に採用しない）

| 出典の型 | 内容 | 価格のオーダー感 |
|----------|------|------------------|
| ログ／PC監視系（ITトレンド掲載の例） | Eye"247" Work Smart Cloud — PC台数別 **月額（1台あたり）** | **約400〜1,100円/台/月**（台数帯により変動） |
| テレワーク／監視系（ベンダー公開） | Watchy — **100円/PC/月・機能単位** など | **100円/PC/月** を下限の参考にしつつ、機能積み上げ型 |
| 勤怠SaaS（ユーザー単価の例） | 勤太くん — **ユーザーあたり月額 200〜300円** など | **低単価帯**は「別カテゴリ（勤怠）」として切り離して説明 |

※ URLは検索・比較サイト経由のため、**営業時は各社の公式料金ページを再確認**すること（エビデンスの鮮度）。

### 3.2 自社プロダクト固有の根拠（社内）

| 根拠 | 内容 |
|------|------|
| 実装 | `monitor.py` 等による **操作・ブラウザ文脈の取得**、**バッチ送信**、**スクリプト実行** — 「1機能だけ」よりスコープが広い。 |
| コスト構造 | クラウド保存・解析・サポートを載せると **原価率** が上がる → **過度に低単価にしない**方が持続可能。 |

### 3.3 反証・限界（正直に書く）

| リスク | 意味 |
|--------|------|
| **無料・超安競合**（例: 海外製の無料プラン等） | 価格だけでは勝てない。**セキュリティ・法務・国内対応・導入支援**で差別化が必要。 |
| **「監視」への抵抗** | 単価以前に **失注** する。同意・目的の明文化が前提（`gtm_aris_project_sales_reverse.md`）。 |
| **まだ有償実績がない** | 上記の数字は **仮説**。初契約の **実際の支払額** が最強のエビデンスになる。 |

## 4. 商品化の方法（パッケージの中身）

| プラン | 含めるもの（例） | 含めない／オプション（例） |
|--------|------------------|----------------------------|
| **パイロット** | エージェント配布、サーバ接続設定、**週次レビュー**、成功基準レポート | 契約スコープ外の追加法人、24/365サポート、カスタム開発 |
| **標準** | 全コア機能、ダッシュボード（ある場合）、**月次レポート** | オンプレ、SI連携はオプション |
| **エンタープライズ** | SLA、専用窓口、**ログ保全期間延長**、稟議用資料テンプレ拡充 | 個別見積 |

**計測**: パイロット終了時に **継続意向（5段階）** と **本契約への移行率** を記録 → 価格改定の入力。

## 5. 検証方法（次の90日）

1. **3社**に **同一条件のパイロット提案**（例: 49万/3ヶ月・**対象組織の定義付き**）を出し、**受注・失注理由**を記録。  
2. 本契約の **企業あたり月額（帯）** を提示したときの **反応**（高い／妥当／機能で勝負したい）。  
3. **粗利**試算: サポート工数＋クラウド原価を1社あたり積み、**下限の月額（企業あたり）**を決める。

## 6. Bossへの依頼（承認が要る場合）

- パイロットの **定価（税込/税別）** と **契約スコープ（対象法人・従業員レンジ等）** の最終決定。  
- 「この価格でなければ受注しない」**下限ライン**の合意（**PC台数課金は採用しない**）。
